# Критика плана Фазы 2 — вердикт по каждому пункту

**Формат:** №. Тезис → вердикт. Для «польза» — доказательство и решение.

---

## 1. A1.1 — три функции вместо одной → ПОЛЬЗА

`is_wb_url`, `is_ozon_url` — не вызываются нигде, кроме гипотетического использования. `validate_competitor_url` сам возвращает платформу или None. Две лишние функции — мёртвый код на старте.

**Решение:** Одна функция `detect_platform(url: str) -> str | None`. Применить.

---

## 2. A3.1 — beautifulsoup4 ради fallback-парсинга → ПОЛЬЗА

Основной путь (`__NEXT_DATA__`) содержит все данные. Если JSON не найден — страница либо не загрузилась (403/блокировка), либо это не страница товара. Fallback на `og:title` через BeautifulSoup вытащит заголовок и больше ничего — этого недостаточно для анализа конкурента (нужны описание, характеристики). Зависимость добавляется ради сценария, который не даёт полезного результата.

**Решение:** Убрать beautifulsoup4. Fallback — пустая строка. Не смогли извлечь JSON → не извлекли ничего. Применить.

---

## 3. B.2 — флаг MEMORY_STORAGE=1 → ПОЛЬЗА

У продукта ноль пользователей. MVP на MemoryStorage — прототип. Поддерживать оба режима (Memory + SQLite) — удвоение поверхности тестирования без причины. SQLite — дефолт, без флага.

**Решение:** Убрать `MEMORY_STORAGE`. Переключиться на SQLiteStorage без обратной совместимости. Применить.

---

## 4. B.3 — фоновая задача очистки copy_cache с TTL 1 час → ПОЛЬЗА (с уточнением)

Критика «преждевременная оптимизация» — неверна для SQLite. В in-memory версии FIFO-вытеснение на 1000 записей решает проблему. В SQLite rows накапливаются вечно. Очистка нужна. Но подход (фоновая asyncio-задача) избыточен.

**Решение:** TTL-проверка на чтении: `SELECT text FROM copy_cache WHERE key = ? AND created_at > datetime('now', '-1 hour')`. Удалили старую запись при промахе. Без фоновой задачи. Применить.

---

## 5. B.4 — 4 параметра конфига → ПОЛЬЗА

`MEMORY_STORAGE` удалён (п.3). `COPY_CACHE_TTL` заменён на проверку при чтении (п.4). Остаются `SQLITE_PATH` и `FSM_STATE_TTL` — оба нужны. Два вместо четырёх.

**Решение:** Оставить `SQLITE_PATH` и `FSM_STATE_TTL`. Применить.

---

## 6. Anti-bot / JS-rendering отсутствует → ПОЛЬЗА (критическое)

WB с 2025 отдаёт пустую оболочку без JS-рендеринга на части запросов. Ozon под Cloudflare. План предполагает `httpx GET` достаточным — это не так. Без признания ограничения фича выглядит как «работает всегда», а по факту — «иногда».

**Решение:** Добавить фазу A0 «Anti-bot стратегия» с явным описанием подхода и ограничений. httpx с desktop User-Agent + заголовками как браузер. WB отдаёт `__NEXT_DATA__` на холодном рендере в ~70% случаев (без JS можно получить). Ozon тяжелее — Cloudflare. При ошибке → fallback на текст. Принять ограничение как осознанное: «парсинг работает не всегда, fallback на копипаст всегда доступен». Применить.

---

## 7. Нет интеграционных тестов парсинга → КОД ДЛЯ КОДА

План — что строить, не как тестировать. Фаза 1 имела 16 тестов в отдельном плане верификации (`2026-05-18-верификация-mvp.md`), не в плане фичи. Структурный тест (мок HTML → assert поля) — полезен, но это критерий задачи, а не отдельная фаза.

**Решение:** Добавить в критерий A1.3: «юнит-тест с мок-HTML, содержащим `__NEXT_DATA__` → assert `wb_title` не пуст». Без отдельной фазы тестирования. Применить частично (в критерий).

---

## 8. Порядок B → A → C вместо A → B → C → ПОЛЬЗА

Персистентность — фундамент. Если делать URL-парсинг первой, состояние FSM при рестарте теряется — ровно то, что B чинит. Плюс: если B сделана первой, A получает персистентность бесплатно.

**Решение:** Порядок: B (SQLite) → A (URL-парсинг) → C (Rate limiting). Применить.

---

## 9. Нет фазы обработки таймаутов URL-фетча → ПОЛЬЗА

План: таймаут 15 секунд httpx. Бот крутит анимацию → 15 секунд ожидания → ошибка → 5 сообщений. Худший UX: долгое ожидание с отрицательным результатом. Пользователь ждал анализ конкурента, получил базовую генерацию.

**Решение:** Укоротить таймаут фетча до 5 секунд. При ошибке — мгновенный fallback без 15-секундного ожидания. Асинхронный фетч: `gather(generate_card, fetch_and_parse)` — если фетч упал, основная генерация не ждёт. Применить.

---

## 10. Критерий «сессии не потеряны» размыт → ПОЛЬЗА

«Сессии не потеряны» — не определено. Что проверять: FSM-состояние восстановилось? Пользователь может продолжить с того же шага? Или просто нет краша?

**Решение:** Конкретный критерий: «Пользователь на шаге competitor_input → рестарт бота → пользователь отправляет текст конкурента → бот принимает и запускает генерацию». Применить.

## 11. Критерий «запускается под Windows и Linux» → ПОЛЬЗА

Это правда уже сейчас. Критерий Фазы 1, выполнен. Создаёт иллюзию объёма.

**Решение:** Удалить из критериев Фазы 2. Применить.

---

## 12. Критерий «длина HTML > 5000» бессмыслен → ПОЛЬЗА

403 Forbidden с капчей → HTML 8000 символов → критерий пройден, данные не извлечены. Количественный критерий не проверяет качество.

**Решение:** Заменить на структурный: «HTML содержит `<script id="__NEXT_DATA__"` (WB) или `<script type="application/ld+json"` (Ozon)». Применить.

## 13. Граничные условия rate limiter не определены → КОД ДЛЯ КОДА

Скользящее окно vs фиксированное, миллисекунды на границе — деталь реализации. План говорит «окно 60 секунд, 3 запроса» — достаточно для понимания что строить. Конкретная реализация окна — решение разработчика на этапе кода, не плана.

**Решение:** Отклонить.

---

## 14. IP-бан маркетплейсом → ПОЛЬЗА

После 3-5 запросов с одного IP WB банит IP на часы. User-Agent не спасает — это TCP-уровень. Четвёртый пользователь за день получает «заблокирован», хотя ничего не нарушал. Не баг — архитектурное ограничение.

**Решение:** Добавить в риски: «WB банит IP после 3-5 запросов → фича одноразовая на IP». Смягчение: fallback на текст всегда доступен. В будущем — прокси-ротация. Применить.

---

## 15. analyze_competitor() раздувается → ПОЛЬЗА

Сейчас функция делает одно: текст → AI → строка. После A1.4 будет: URL → HTTP → HTML → JSON → AI. Три уровня абстракции, два сетевых. Тестировать изолированно невозможно.

**Решение:** Разделить: `url_fetcher.py` содержит `fetch_product_page()` + `extract_product_text()`. Вызов в хендлере (`_do_generate`), не в `analyze_competitor()`. `analyze_competitor()` остаётся чистым AI-вызовом. Применить.

## 16. Конфликт анимации и долгого фетча → ПОЛЬЗА

Генерация идёт 30+ секунд. Фетч URL с таймаутом 5 секунд — но анимация не знает об ошибке фетча до завершения генерации. Если фетч упал на 3-й секунде, пользователь видит ещё 27 секунд «Собираю правила Ozon...», хотя конкурент уже отвалился.

**Решение:** Параллельный `gather` (п.9) решает: основная генерация и фетч идут одновременно. Фетч упал → competitor_analysis = "". Применить (уже в п.9).

**Решение:** Отклонить.
