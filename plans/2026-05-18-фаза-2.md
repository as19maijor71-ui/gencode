# План: Фаза 2 — От MVP к production (18.05.2026)

## Цель
Три фичи, превращающие прототип в продукт, которым можно пользоваться не только автору:
1. Персистентное хранилище — сессии не пропадают при рестарте
2. Парсинг URL конкурента (WB/Ozon) — вместо ручного копипаста
3. Rate limiting — защита от перебора запросов

## Почему именно эти три и в этом порядке

### Приоритет 1: Персистентное хранилище (фундамент)
Сейчас `MemoryStorage` — всё теряется при перезапуске бота. Архитектура (`08-nonfunctional/риски-архитектуры.md`) прямо говорит: «В Фазе 2 — PostgreSQL». SQLite как промежуточный шаг: ноль зависимостей, один файл, можно мигрировать на PostgreSQL позже. **Делается первой**, потому что фичи A и C работают поверх неё — состояние URL-фетча, кэш копирования и лимиты (опционально) получают персистентность бесплатно.

### Приоритет 2: URL-парсинг конкурента
Архитектура (`02-core/главный-цикл.md`, Режим B) явно описывает: «Селлер даёт URL конкурента. AI извлекает данные по ссылке, сравнивает, находит точки обхода». Сейчас селлер вынужден копировать текст вручную — это компромисс MVP, не финальное решение. Парсинг URL — это то, что отличает продукт от ChatGPT «в лоб».

### Приоритет 3: Rate limiting
Без лимитов любой может заспамить бота и сжечь бюджет на AI. Архитектура (`08-nonfunctional/безопасность.md`): «В Фазе 2 — лимиты». Простейший in-memory rate limiter на старте.

---

## Критерий «Фаза 2 готова»

1. Перезапуск бота → пользователь на шаге `competitor_input` отправляет текст → бот принимает и запускает генерацию (FSM-состояние восстановилось из SQLite)
2. Бот принимает URL WB/Ozon → парсит содержимое → 6-е сообщение с анализом. Битый URL → 5 сообщений без краша
3. 3 генерации от одного пользователя за 60 секунд → 4-я блокируется с сообщением «Слишком много запросов»
4. **Регрессия:** базовый flow (без конкурента), копипаст конкурента текстом, кнопка «Копировать всё» — не сломаны

### Фича A: URL-парсинг конкурента — порядок 2-й

#### Фаза A0: Anti-bot стратегия и ограничения

**Проблема:** WB с 2025 отдаёт пустую оболочку без JS-рендеринга на части запросов. Ozon под Cloudflare. `httpx GET` не гарантирует получение данных.

**Подход:**
- Desktop User-Agent (Chrome Windows), заголовки `Accept-Language: ru`, `Referer: https://www.google.com`
- WB: `__NEXT_DATA__` доступен на холодном рендере в ~70% случаев без JS
- Ozon: `application/ld+json` + мета-теги — ниже вероятность успеха
- Таймаут фетча: **5 секунд** (не 15 — плохой UX при долгом ожидании с отрицательным результатом)
- При любой ошибке → fallback на пустую строку, пользователь видит «⚠️ Не удалось загрузить карточку. Генерирую без анализа.» и 5 сообщений
- Копипаст текстом остаётся всегда доступным — это не деградация, а альтернативный путь

**Ограничение (принято осознанно):** парсинг работает не в 100% случаев. Это не баг — это архитектурное ограничение httpx-подхода. В будущем: прокси-ротация или scraping API. Селлер всегда может скопировать текст вручную.

#### Фаза A1: Ядро парсинга

- [x] **A1.1 URL-валидатор**
  - `cardgen/engine/url_fetcher.py` — `detect_platform(url: str) -> str | None`
  - Возвращает `"wb"`, `"ozon"` или `None`
  - Паттерны: `wildberries.ru/catalog/...`, `ozon.ru/product/...` (включая `www.`, `https://`, мобильные поддомены)
  - **Критерий:** 5 валидных URL → `"wb"`/`"ozon"`. 5 левых (google.com, vk.com, текст) → `None`

- [x] **A1.2 HTTP-фетчер**
  - `fetch_product_page(url: str) -> str` — httpx GET, таймаут 5с
  - User-Agent: десктопный Chrome Windows. Заголовки: `Accept-Language: ru`, `Referer: https://www.google.com`
  - Обработка: таймаут / 403 / 429 → райз `CompetitorFetchError`. 404 → райз с сообщением «Карточка не найдена»
  - **Критерий:** валидный URL WB → HTML содержит `"__NEXT_DATA__"`. Валидный URL Ozon → HTML содержит `"application/ld+json"`

- [x] **A1.3 Извлечение текста из HTML**
  - `extract_product_text(html: str, platform: str) -> str`:
    - WB: `json.loads(re.search(r'<script id="__NEXT_DATA__"[^>]*>(.*?)</script>', html))` → извлечь title, description, characteristics из `props.pageProps.product`
    - Ozon: `json.loads(re.search(r'<script type="application/ld+json">(.*?)</script>', html))` → `name`, `description`
  - Обрезка до `COMPETITOR_MAX_LENGTH` (3000 символов)
  - Ошибка парсинга JSON → пустая строка (без fallback-зависимостей)
  - **Критерий:** юнит-тест с мок-HTML, содержащим `__NEXT_DATA__` → `extract_product_text` возвращает непустую строку с названием товара. Живой URL WB → непустой текст

- [x] **A1.4 Интеграция в хендлер (не в генератор)**
  - Фетч и парсинг — в хендлере (`_do_generate`), **не** внутри `analyze_competitor()`
  - `analyze_competitor()` остаётся чистым AI-вызовом: текст → AI → строка
  - Разделение: `url_fetcher.py` знает про HTTP и HTML. `generator.py` знает только про AI
  - **Критерий:** `analyze_competitor()` не импортирует httpx и не принимает параметр `competitor_url`

#### Фаза A2: Интеграция в бота

- [x] **A2.1 FSM — competitor_input принимает URL**
  - Валидация: если текст сообщения начинается с `http` и проходит `detect_platform()` → URL
  - Если URL валидный → бот показывает «🔗 Анализирую карточку конкурента по ссылке...»
  - Если URL невалидный → «❌ Это не ссылка на товар WB или Ozon. Скопируй URL из адресной строки.»
  - Если сообщение — не URL → поведение без изменений (текст конкурента, как сейчас)
  - Текст приглашения дополнен: «Отправь URL карточки WB или Ozon — или скопируй текст вручную.»
  - **Критерий:** flow URL WB → парсинг → 6 сообщений. Flow текст (как раньше) → без изменений. Flow левый URL → сообщение об ошибке, пользователь может повторить

- [x] **A2.2 _do_generate() — параллельный фетч**
  - Если `competitor_url` не пуст → `asyncio.gather(generate_card(...), fetch_and_parse(url), return_exceptions=True)`
  - `fetch_and_parse()` внутри: `fetch_product_page()` → `extract_product_text()` → `analyze_competitor(text=extracted)`
  - Если `fetch_and_parse` упал → `competitor_analysis = ""`, 5 сообщений + предупреждение «⚠️ Не удалось загрузить карточку. Генерирую без анализа.»
  - **Ключевое:** генерация и фетч идут параллельно. Пользователь не ждёт 30+5 секунд — ждёт max(30, 5) = 30 секунд
  - **Критерий:** битая ссылка → 5 сообщений (не краш). Живая ссылка → 6 сообщений. Время генерации с живой ссылкой ≈ времени генерации без ссылки (параллельность работает)

### Фича C: Rate limiting — порядок 3-й

- [x] **C.1 In-memory rate limiter**
  - `cardgen/bot/rate_limiter.py` — класс `RateLimiter`:
    - `check_and_record(user_id: int) -> bool` — записывает timestamp текущего запроса, возвращает False если лимит превышен
    - Окно: 60 секунд, максимум 3 запроса в окне (скользящее — по timestamp-ам)
    - Хранилище: `dict[int, list[float]]`, очистка старых timestamp-ов при каждом вызове
  - **Критерий:** 3 вызова `check_and_record` подряд → True. 4-й → False. Вызов через 61 секунду после первого → True

- [x] **C.2 Мидлварь**
  - `RateLimitMiddleware` (aiogram middleware) — перехватывает сообщения на `product_input` и `competitor_input`
  - При превышении → ответ «⚠️ Слишком много запросов. Подожди 60 секунд.»
  - **Не блокирует:** `/start`, «Начать заново», инлайн-кнопки
  - **Критерий:** 4 описания товара подряд → 4-е блокируется. «Начать заново» → не блокируется

## Риски

| Риск | Уровень | Смягчение |
|---|---|---|
| WB/Ozon меняют структуру `__NEXT_DATA__` | High | Парсинг конкретных полей, а не всей структуры. При поломке — fallback на пустую строку (без краша). Юнит-тест с мок-HTML ловит поломку до деплоя |
| WB банит IP после 3-5 запросов → фича одноразовая на IP | High | Fallback на копипаст текста всегда доступен. В будущем — прокси-ротация или scraping API. Ограничение принято осознанно (см. A0) |
| Ozon под Cloudflare — httpx не проходит | High | Таймаут 5с → быстрый fallback. Не тратим 15 секунд на заведомо безнадёжный запрос |
| Конфликт анимации и долгого фетча | Medium | Параллельный `gather` — анимация крутится только пока идёт генерация, фетч не удлиняет ожидание |
| SQLite под Docker — путь к файлу | Low | `cardgen/data/` — volume в docker-compose, директория создаётся при старте |
| SQLite не держит конкурентную запись | Low | aiogram исполняет хендлеры последовательно для одного пользователя. WAL-режим. Для <50 пользователей достаточно |
| Переезд с MemoryStorage на SQLite — слом FSM | Low | Нет обратной совместимости (ноль пользователей). Просто переключить. Старые сессии в памяти — потеряны при рестарте, новых нет |

## Порядок реализации

1. **B1-B3** — SQLite хранилище (фундамент)
2. **A0-A2** — URL-парсинг (главная пользовательская фича)
3. **C1-C2** — Rate limiting (защита)
4. **Конфиг** — все переменные разом

## Что НЕ входит в Фазу 2

- Webhook (polling держит до 500 пользователей — `08-nonfunctional/риски-архитектуры.md`)
- Авторизация (отдельная большая фича — OAuth/Telegram-логин, личный кабинет)
- Аналитика использования
- Платёжная система / подписка
- Прокси-ротация / scraping API для URL-парсинга
- Деплой на сервер (отдельный план — нужен VPS, домен, CI/CD)
- Push в удалённый репозиторий (отдельная задача)
